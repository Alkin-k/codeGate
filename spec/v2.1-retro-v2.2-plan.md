# V2.1 复盘结论与 V2.2 改造清单

> 日期: 2026-04-24
> 项目: CodeGate
> 范围: 仅基于当前项目的 benchmark、源码和已落地 ADR 复盘

## 1. 文档目的

这份文档只回答三个问题：

1. V2.1 到底证明了什么
2. V2.1 还没有解决什么
3. V2.2 应该优先改什么，什么先不要做

它不是 benchmark 原始结果表，也不是 roadmap 全量规划，而是面向当前版本演进的收敛文档。

## 2. V2.1 一句话结论

V2.1 证明了 CodeGate 已经从“gate 偏松的流程原型”进入“具备初步拦截能力的治理 MVP”，
但还没有进入“可对外强证明”的阶段。

更具体地说：

- V2.0 的核心问题不是链路跑不通，而是放行语义偏松
- V2.1 已经把 `assumed_defaults`、`unresolved_items`、`high-risk` 这些信号接入 deterministic policy
- `case_3` / `case_4` 进入修订轮，说明系统开始具备真实的“先拦后修”能力
- `case_5` 不再出现 V2.0 那种“roles 表缺失却仍 approve”的误放行
- 但当前 benchmark 仍然主要证明“治理链自洽”，还没有充分证明“真实工程场景下的稳定治理价值”

## 3. V2.1 已验证的结论

### 3.1 Gate 语义已经收紧

V2.1 最重要的进步，不是最终仍然 `5/5 approve`，而是“什么情况下不能直接 approve”已经更清晰了。

当前已进入 deterministic policy 的信号包括：

- blocking findings
- drift / coverage 阈值
- unresolved_items
- assumed_defaults 违规
- high-risk 任务的更严门槛

这意味着 CodeGate 已经不再只是 advisory reviewer，而开始像真正的 gate。

### 3.2 risk_level 已从展示字段变成策略字段

V2.0 的 `risk_level` 只是在 state 中流转，并不改变行为。
V2.1 已经让 high-risk 任务使用更严的 `drift` / `coverage` 阈值，并对高风险场景增加了人工升级条件。

这是正确方向，因为治理系统的价值不在于“所有任务一刀切走同样流程”，
而在于“不同风险任务有不同的通过标准”。

### 3.3 V2.0 最大误放行样本已被修正

`case_5_migration` 是 V2.0 最大的反例：

- contract 假定存在独立 `roles` 表
- reviewer 发现了 roles 表缺失
- 但最终仍然 approve

V2.1 里这个问题被真正修掉了：

- contract 明确要求 `roles(id, name)` 和 `user_roles(user_id, role_id)`
- execution 也实际产出了 `roles + user_roles + rollback`
- 最终 approve 才重新成立

这说明 V2.1 的进步不是“分数更好看”，而是至少修掉了一个关键的治理漏洞。

### 3.4 benchmark 的治理成本表达更接近真实问题

当前 benchmark 已经能把治理层成本和 executor 成本拆开看：

- governance overhead
- executor time
- governance tokens
- iterations

这让 V2.1 的结果具备了“分析治理层本身”的基础，而不是继续把执行器延迟误算成治理成本。

## 4. V2.1 暴露出的核心问题

### 4.1 Spec Council 质量仍然不稳定

V2.1 最大的未解问题，已经不再只是 gate，而是 contract 自身质量。

典型例子是 `case_2_auth`：

- `non_goals` 写了“不修改现有 users 表结构（不添加 password 字段）”
- `constraints` 又写了“password 字段需新增（因原表无此字段，需迁移）”
- execution 实际也新增了 migration
- reviewer / gate 最终仍给出 `approve`

这说明当前系统缺少一个 **contract consistency validator**。
也就是说，Spec Council 虽然能产出结构化 contract，但还不能保证 contract 自身无冲突。

这会带来两个严重后果：

- executor 被交给了一份自相矛盾的合同
- reviewer 很可能在“执行符合一半 contract”时仍给出高分

### 4.2 ExecutionReport 仍然可能不具备可审计性

当前 `ExecutionReport` schema 要求提供 `code_output`，
但系统没有校验 `code_output` 是否真的是可审计实现。

典型例子是 `case_2_auth`：

- `code_output` 只有一句占位文本：`The implementation is provided in the file_list files.`
- reviewer 仍给出 `0 findings`
- gate 最终直接 `approve`

这说明当前 reviewer 有时并没有真正“审代码”，而是在审 executor 的自报摘要。

这会伤到 CodeGate 最核心的价值主张：

> 它本应审查实现是否符合 contract，而不是审查一段自我陈述是否看起来合理。

### 4.3 迭代证据链仍然不完整

当前 benchmark 和 artifact store 主要保存的是最终态：

- 最终 contract
- 最终 execution_report
- 最终 review_findings
- 最终 gate_decision

但对于 `case_3` / `case_4` 这类“首轮被拦截、二轮修复通过”的任务，
系统没有把每一轮 evidence 单独保存成可复盘产物。

这带来一个很现实的问题：

- 文档里可以写“首轮被正确拦截”
- 但 run 目录里无法直接拿出对应证据链证明这一点

对于治理产品来说，这会削弱“拦截价值”的可展示性。

### 4.4 contract 中仍有死字段

`required_tests`、`rollback_conditions` 在 schema 中已经存在，
但目前还没有真正进入 deterministic policy。

这意味着：

- contract 写了必要测试，不代表 gate 会检查它们
- contract 写了 rollback 条件，不代表高风险任务真的会因此更严格

这些字段现在更像“文档丰富度字段”，还不是“治理生效字段”。

### 4.5 benchmark 的可比性仍不够强

V2.1 比 V2.0 更好了，但当前版本对比还不是完全 apples-to-apples。

原因有两个：

1. 每次 rerun 时，Spec Council 可能会生成不同 contract
2. benchmark 结果里最终态和首轮态混在一起叙述

这会导致一个问题：

你看到的版本改进，可能同时来自：

- gate 语义收紧
- reviewer prompt 变严
- contract 本身也变了

所以当前 benchmark 更适合做“方向判断”，还不适合做“强归因证明”。

### 4.6 executor 仍然是模拟器

当前 `BuiltinLLMExecutor` 仍然不产出真实 patch，而是产出文本化 implementation report。

这意味着当前 benchmark 仍然主要验证：

- 治理流程是否成立
- prompt / policy / state machine 是否能闭环

它还没有验证：

- 面对真实 patch / diff 时 reviewer 是否稳定
- 面对真实 repo 变更时 gate 是否可靠
- 治理层与真实 coding agent 组合后是否仍然有效

## 5. 对 V2.1 的总体判断

### 5.1 V2.1 已经不再是“空转治理”

这一版最大的积极信号是：

- 系统开始真的拦截
- 修订循环开始有意义
- `risk_level` 开始改变策略
- V2.0 最大误放行样本被修正

所以 V2.1 不是“又多了一版 prompt 调优”，而是治理语义确实向前走了一步。

### 5.2 但 V2.1 仍然不该被包装成“已完全可推广”

原因不是功能不够多，而是以下几件事还没补齐：

- contract 自身一致性还不能保证
- reviewer 仍可能在没有可审代码时给出高分
- 迭代拦截证据还不能完整展示
- benchmark 可比性和可复现性还不够强
- 真实 executor 还未接入

所以 V2.1 更合适的定义是：

> 治理语义已基本成型，但还需要一轮“证据链强化 + contract 质量强化 + benchmark 方法学强化”。

## 6. V2.2 的版本目标

V2.2 不追求做更多功能，而追求把以下四件事做实：

1. contract 在执行前就是可信的
2. approve 必须基于可审计实现证据
3. 每一轮 revise 都能完整复盘
4. benchmark 能稳定比较版本改进，而不是混杂多重变量

一句话说：

> V2.2 要把 CodeGate 从“会拦的治理 MVP”推进到“能证明自己为什么拦、为什么放”的治理内核。

## 7. V2.2 改造清单

### P0：本期必须完成

#### 1. Contract Consistency Validator

目标：

- 在 `spec_approved` 前增加 contract 自检
- 检查 `goals / non_goals / constraints / assumed_defaults / rollback_conditions` 是否自相矛盾
- 检查 must acceptance criteria 是否与 non_goals 冲突
- 检查高风险任务是否缺少 rollback / risk 信息

建议动作：

- 新增 validator 模块，输出结构化 `contract_violations`
- validator 失败时不进入 executor，而是转 `revise_spec`
- benchmark 记录 `spec_validated=true/false`

验收标准：

- `case_2_auth` 这类“non_goal 与 constraint 冲突”的 contract 必须在执行前被拦住

#### 2. ExecutionReport Hardening

目标：

- 防止 reviewer 在没有真实实现证据时给出 `approve`

建议动作：

- 为 `ExecutionReport` 增加“最小可审计证据”要求
- 至少满足以下之一：
  - 真实 patch / diff
  - 完整代码实现
  - 可打开的 artifact 引用
- 拒绝占位式 `code_output`
- 如果证据不足，直接 `revise_code` 或 `escalate_to_human`

验收标准：

- 像 `The implementation is provided in the file_list files.` 这种占位输出不能再进入正常 approve 流程

#### 3. Per-Iteration Evidence 持久化

目标：

- 让“首轮被拦截、二轮修复通过”成为可审计事实，而不是文档叙述

建议动作：

- 每一轮单独落盘：
  - `iteration_1/execution_report.json`
  - `iteration_1/review_findings.json`
  - `iteration_1/gate_decision.json`
  - `iteration_1/policy_result.json`
- 空 findings 也必须持久化为 `[]`
- 保留 gatekeeper 给 executor 的反馈内容

验收标准：

- `case_3` / `case_4` 的首轮拦截可以直接从 run 目录中复盘出来

#### 4. 让死字段真正进入治理

目标：

- 让 contract 中的 `required_tests`、`rollback_conditions` 不再只是装饰字段

建议动作：

- 对 `required_tests` 增加检查策略：
  - 缺少测试说明或未声明验证结果时不能直接 approve
- 对高风险任务的 `rollback_conditions` 增加最低要求：
  - 缺失时不允许直接 spec approve
  - 或在 gate 环节强制 revise / escalate

验收标准：

- contract 里写了必要测试和回滚条件，就必须对最终 gate 有实质影响

#### 5. Benchmark Snapshot Freeze

目标：

- 提高版本对比的可比性

建议动作：

- 为 benchmark case 固定 task fixture
- 为 approved contract 保存基线快照
- 报告中显式区分：
  - first-round metrics
  - final metrics
- 记录 run metadata（版本、prompt 版本、policy 版本、fixture 版本）

验收标准：

- V2.1 和 V2.2 的对比能够明确说明“改进来自哪里”

### P1：高优先，但可在 P0 之后完成

#### 6. Reviewer / Gate Score Normalization

目标：

- 减少 LLM “感觉打分”带来的漂移

建议动作：

- reviewer 先输出 findings
- 系统再根据 findings / unmet criteria / unmet assumptions 进行分数归一化
- gate 优先消费归一化后的分数，而不是原始自由打分

收益：

- `drift_score` / `coverage_score` 更接近可解释指标

#### 7. Contract 质量指标进入 benchmark

目标：

- benchmark 不只量“代码治理质量”，也量“spec 质量”

建议动作：

- 新增指标：
  - contradiction_count
  - assumed_defaults_count
  - spec_revision_count
  - validator_failures

收益：

- 能区分“问题出在 spec”还是“问题出在 execution”

#### 8. 接入一个真实 executor adapter

目标：

- 让 V2.2 开始摆脱“纯模拟 executor”的上限

建议动作：

- 只接一个真实 executor
- 优先完成最薄适配，不在 adapter 层做复杂智能编排
- 要求返回真实 patch / artifact 引用

验收标准：

- 至少 1 个真实 repo 上跑通完整闭环

#### 9. PR / GitHub 落点

目标：

- 让 CodeGate 的输出进入真实研发工作流

建议动作：

- 输出 contract summary
- 输出 review findings
- 输出 gate decision
- 输出 iteration history

收益：

- 从“本地 benchmark 工具”向“团队可感知的治理层”迈进一步

### P2：后续增强项

#### 10. 风险分级驱动的流程分叉

可以考虑但不急于在 V2.2 完成：

- low-risk 任务走更快路径
- high-risk 任务强制人工 approve
- risk_level 由 Spec Council 自动评估，而不是完全外部输入

#### 11. 多 reviewer 角色化

在当前单 reviewer 模式稳定后，再拆分：

- correctness reviewer
- security reviewer
- performance reviewer
- maintainability reviewer

这项更适合放在“证据链和 contract 质量已经稳定之后”。

## 8. V2.2 非目标

V2.2 明确不做以下事情：

- 不做完整 Web UI
- 不做很多 adapter
- 不做复杂 agent 编排平台
- 不把项目转成大而全的企业研发平台

V2.2 的重点只有一个：

> 把治理内核的可信度、证据链和 benchmark 方法学补扎实。

## 9. 建议的实施顺序

建议按下面顺序推进：

1. Contract Consistency Validator
2. ExecutionReport Hardening
3. Per-Iteration Evidence 持久化
4. required_tests / rollback_conditions 进入 policy
5. Benchmark Snapshot Freeze
6. Reviewer / Gate Score Normalization
7. 真实 executor adapter
8. GitHub / PR 落点

这样做的原因是：

- 先补 contract 质量，避免后面在坏 contract 上做更多治理
- 再补证据链，避免“改进了但无法证明”
- 再补 benchmark 方法学，避免版本比较失真
- 最后再接真实 executor 和外部工作流落点

## 10. V2.2 完成后的验收标准

V2.2 不要求一口气解决所有问题，但至少要满足以下 5 条：

1. 自相矛盾的 contract 在执行前就会被拦下
2. 没有真实实现证据的 execution_report 不能直接通过 gate
3. `case_3` / `case_4` 这类修订任务可以完整复盘每一轮 evidence
4. `required_tests` 和 `rollback_conditions` 对最终 gate 有实质影响
5. benchmark 报告能区分首轮表现和最终表现，并能稳定复现实验条件

如果这 5 条完成，V2.2 就算达成目标。

## 11. 最终判断

V2.1 的意义是把 CodeGate 从“会跑的治理流程”推进到了“开始会拦的治理系统”。
V2.2 的意义则应该是：

> 把 CodeGate 从“开始会拦”推进到“拦得有依据、放得有证据、版本对比有说服力”。

这是当前版本演进里最应该优先完成的一步。
