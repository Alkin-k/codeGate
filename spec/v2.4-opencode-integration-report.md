## 2. OpenCode Adapter 架构

```
┌───────────────────────────────────────────────────────────┐
│                CodeGate Governance Pipeline                │
│                                                           │
│   SpecCouncil ─→ Executor ─→ Reviewer ─→ Gatekeeper     │
│                     │                                     │
│                     ▼                                     │
│              ┌─────────────┐                             │
│              │ Adapter Layer│                             │
│              └──────┬──────┘                             │
│                     │                                     │
│         ┌───────────┼───────────┐                        │
│         ▼           ▼           ▼                        │
│   BuiltinLLM    OpenCode    (future)                     │
│   (模拟文本)    (真实代码)   OMO/Claude                    │
└───────────────────────────────────────────────────────────┘
```

### 2.1 调用流程

```
1. CodeGate 将 ImplementationContract 序列化为 markdown prompt
2. subprocess 调用:
   opencode run "<prompt>" --format json --dir <sandbox> \
     [--dangerously-skip-permissions]  # 仅 sandbox 模式
     --model <model>
3. 解析 JSON 事件流，提取：
   - tool_use (type=write) → 文件路径 + 内容
   - text → 总结文本
   - step_finish → token 消耗
4. 文件系统扫描（mtime 比对）兜底检测遗漏变更
5. 构造 ExecutionReport（files_content + file_list + code_output）
```

### 2.2 opencode JSON 事件格式

```json
// 文件写入事件
{"type":"tool_use","part":{"tool":"write","state":{
  "status":"completed",
  "input":{"filePath":"/path/to/file.py","content":"print('hello')"},
  "output":"Wrote file successfully."
}}}

// 文本总结事件
{"type":"text","part":{"text":"Done. Created `hello.py`."}}

// Token 统计事件
{"type":"step_finish","part":{"tokens":{"total":22956,"input":18516,"output":88}}}
```

## 3. 端到端验证结果

### 3.1 Case 1 Fibonacci — BuiltinLLM vs OpenCode

| 维度 | BuiltinLLM (V2.3.2) | OpenCode (V2.4) |
|------|---------------------|-----------------|
| **executor** | builtin_llm | opencode |
| **model** | deepseek-chat | kimi-for-coding/k2p6 |
| **产出物** | 文本模拟（JSON 内的字符串） | **真实 Python 文件** |
| **file_list** | `['fib.py']` | `['fib.py']` |
| **files_content** | `{}` (未填充) | `{'fib.py': '...'}` ← **有内容** |
| **代码质量** | 无 docstring、无类型标注 | ✅ 有 docstring + 类型标注 + assert 测试 |
| **自验证** | ❌ 无 | ✅ opencode 自己跑了 assert 验证 |
| **exec tokens** | 907 | 202,427 |
| **exec 耗时** | 2.7s | 93.0s |
| **governance overhead** | 11.1s | 11.1s ← **治理层耗时不受 executor 影响** |
| **reviewer findings** | 1 (assumed_defaults P1) | 0 |
| **gate decision** | revise_code (Rule 7 override) | ✅ **approve** |
| **drift** | 0 | 0 |
| **coverage** | 100 | 100 |

### 3.2 关键发现

1. **治理层耗时独立于 executor**：两种 adapter 的 governance overhead 都是 ~11s，证明治理层是真正解耦的
2. **opencode 产出质量更高**：生成代码包含 docstring、类型标注、测试断言
3. **opencode 自带验证**：执行过程中自动跑了 `assert fib(0)==0` 等测试
4. **代价是 token 消耗和时间**：opencode 消耗 202K tokens、93s，vs builtin_llm 的 907 tokens、2.7s
5. **files_content 字段生效**：首次在 evidence 中记录了结构化的文件内容

### 3.3 OpenCode 生成的实际代码

```python
def fib(n: int) -> int:
    """Calculate the nth Fibonacci number iteratively.
    
    Args:
        n: Non-negative integer index in the Fibonacci sequence
        
    Returns:
        The nth Fibonacci number (F(0)=0, F(1)=1)
    """
    if n == 0:
        return 0
    if n == 1:
        return 1
    
    prev2, prev1 = 0, 1
    for _ in range(2, n + 1):
        current = prev2 + prev1
        prev2, prev1 = prev1, current
    
    return prev1


if __name__ == "__main__":
    assert fib(0) == 0, "fib(0) should return 0"
    assert fib(1) == 1, "fib(1) should return 1"
    assert fib(2) == 1, "fib(2) should return 1"
    assert fib(10) == 55, "fib(10) should return 55"
    assert fib(20) == 6765, "fib(20) should return 6765"
    print("All tests passed!")
```
